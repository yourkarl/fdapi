# fdapi SDK API 设计评审意见（架构师视角）

> 2026-07-14 · 评审依据：`dts-sdk.d.ts`（70 命名空间 / 1242 方法）与 133 篇 API/教程文档的静态分析。未连接运行时，涉及运行时行为的结论均标注"待核实"。
> 文档工程本身经四轮审查已收敛（见《文档站优化建议报告》），本文只谈 **API 设计**。

## 总体评价

fdapi 的能力广度在国产数字孪生 SDK 中属第一梯队：70 个命名空间覆盖 GIS 底座、BIM 单体化、水利/交通/信号/战场仿真、视频融合、3DGS 等，且 50 个对象类支持 `updateBegin/End` 批量更新——这个覆盖率说明团队有过一次成功的接口范式统一。主要问题不在能力，而在**演进机制缺失**与**契约一致性**：API 表面积只增不减、失败语义未定义、对象类接口不可预测。这些问题在人肉查文档的时代是"体验税"，在 AI 辅助编程时代会被放大为"幻觉源"——模型会理所当然地假设 `boxTrigger.show()` 存在。

## 发现与建议（按严重度）

### F1 · 版本化反模式："克隆类"演进 ⚠⚠⚠

证据：`finiteElement/finiteElement2`、`imageryLayer/imageryLayer2`、`vehicle/vehicle2` 三对并存；水动力一域四套（`HydrodynamicModel`、`HydrodynamicModel2`、`hydrodynamic1d`、`hydrodynamic2d`）；`Tag` 靠文档注记"已停止更新，推荐 Marker"退役，但 API 仍在。

这是典型的"没有废弃机制，只能靠克隆新类演进"。每次能力升级都让 API 表面积翻倍，选型成本转嫁给每一个接入者（"我该用 vehicle 还是 vehicle2？"文档没有回答）。

**建议**：一，为每对克隆类补一段"选型指引 + 旧类状态声明"（短期，纯文档）；二，v8 建立正式废弃生命周期（`@deprecated` 标注 → 保留两个次版本 → 移除），新能力优先以 `options` 参数扩展而非克隆类；三，长期目标是 vehicle2 类能力回并主类，旧名作别名（平台已有 `shapeFile/shp → shapeFileLayer` 的别名先例，机制是现成的）。

### F2 · 对象类接口契约不可预测 ⚠⚠⚠

证据：55 个具备 `add` 的对象管理类中，12 个 CRUD 不完整——`boxTrigger` 缺 update/show/hide/focus/get 五件，`cameraTour` 缺五件，`battlefieldSimulation` 缺四件；`show/hide` 各有 8 个类缺失。

开发者（和 AI）从 40 个类学到的直觉在第 41 个类上失效。缺失有些可能是有意的（触发器无所谓显隐？），但"有意"与"遗漏"在 API 表面上不可区分。

**建议**：定义《对象类最小接口契约》——凡具备 `add` 的类必须实现 `add/update/delete/clear/show/hide/focus/get`，确属语义不适用的在文档显式声明"本类不支持 X，因为…"；把契约检查写进 API 评审门禁（可以直接扩展现有 `gen_dts.py` 输出对称性报告，成本一天）。

### F3 · 失败语义是 API 契约空洞，不只是文档缺失 ⚠⚠⚠

证据：`async-call.md` 通篇 0 处提及 reject/catch/失败；全站示例无错误处理。此前文档审查把它归为"文档欠账"，但从架构视角看更可能是 **SDK 层从未定义失败契约**（Promise 是否 reject？错误对象结构？超时行为？——待连运行时核实）。对照组：管理服务侧有完整的 ManageResult 错误码体系，说明团队有能力做，只是 fdapi 层没做。

**建议**：在 SDK 层定义并文档化：何种情况 reject、错误对象标准结构 `{code, message, detail}`、错误码表（可与 ManageResult 风格统一）、超时与断线行为。这是 v7.x 可以做的非破坏性补强——先把"现状实际行为"测清楚写成文档，再在 v8 收敛为承诺。

### F4 · 魔法数字枚举 ⚠⚠

证据：21+ 篇文档存在 `0/1` 数值模式参数（`displayMode: 0水样式 1热力`、`renderMode: 0~3`、`serviceProvider: 1~5` 等），调用方代码里全是裸数字。

**建议**：SDK 导出常量枚举（`fdapi.enums.DisplayMode.Heat`），`dts-sdk.d.ts` 同步把这些参数从 `number` 升级为字面量联合类型（`0 | 1`）——后者只改 `gen_dts.py` 的类型映射，配合参数表"取值范围"抽取即可自动化，IDE 里立刻能拦住非法值。

### F5 · 命名与领域边界 ⚠⚠

证据：`HydrodynamicModel2` 是全部 70 个命名空间里唯一大写开头的异类（文档内两种拼写混用，运行时真名待核实）；`tools`、`plot`、`editHelper` 各自横跨两个文档分类（analysis/events、battle/measure），本质是**一个命名空间承载了多个领域的职责**，文档只能靠双挂载来兜。

**建议**：短期连实例核实 Hydrodynamic 大小写后全站统一；v8 考虑按领域拆分 tools（`analysisTools` / `interactTools`），旧名保留为别名。

### F6 · 全局单例与现代前端工程的张力 ⚠⚠

证据：`DigitalTwinAPI` new 之后挂全局 `fdapi` 调用；而平台自己有多路视频教程（multi-video）——多实例需求真实存在。全局单例与微前端、多标签页、SSR、单页多视口场景冲突，也让单元测试无法并行。

**建议**：v8 让 `new DigitalTwinAPI()` 返回可独立使用的实例句柄，全局 `fdapi` 降级为"第一个实例的兼容别名"。规划中的 `@dts/fdapi-loader` npm 包正是引入这个改变的天然切入点（loader 返回实例，新老写法并存）。

### F7 · 批量更新的命令式风险 ⚠

`updateBegin/End` 覆盖 50 类是亮点，但命令式配对依赖调用方自律，漏掉 `updateEnd` 的后果（更新丢失？状态泄漏？）文档未说明。**建议**：补充漏配对的行为说明；提供函数式语法糖 `fdapi.marker.batch(() => {...})` 自动配对，出错也能 finally 收尾。

## 治理层意见

一，**契约先行，反转生成方向**。现状是"文档 → gen_dts.py 逆向出类型"，文档笔误直接污染类型（water-mesh 事故已验证此风险）。建议中期把 API 定义收敛为一份机器可读契约（JSON Schema/IDL），由它生成文档骨架、d.ts、补全数据、参数面板——文档只写人类叙述部分。现有管线脚本积累正好是迁移的基础。

二，**建立 API 评审门禁**。新增/修改 API 必须过一张 checklist：命名小驼峰、最小接口契约、无裸数字枚举、失败语义声明、废弃标注规范。用 CI 跑对称性/命名检查（F2/F5 的检测脚本本次评审已写出原型）。

三，**把本文的"待核实"项清零**。三件事需要连真实实例各花十分钟：Promise 失败行为、Hydrodynamic 大小写、updateBegin 漏配对后果。建议放进下次联调的 checklist。

## 优先级建议

| 优先级 | 事项 | 破坏性 | 成本 |
|--------|------|--------|------|
| P0 | F3 失败语义测清并文档化；F5 大小写核实统一 | 无 | 天级 |
| P1 | F1 克隆类选型指引；F2 契约检查进 CI；F4 d.ts 字面量类型 | 无 | 周级 |
| P2 | F4 常量枚举导出；F7 batch 语法糖；F6 实例句柄（随 loader 包） | 低（增量） | 迭代级 |
| v8 | F1 废弃生命周期；F5 领域拆分；契约先行管线 | 有（需迁移指南） | 版本级 |
