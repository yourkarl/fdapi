# 坐标系与数据配准

数据"加载出来了但位置不对"，九成是配准问题。本文是接入任何空间数据前的必读检查单。基础的坐标数组格式见[坐标系与坐标转换](/docs/tutorials/coordinates)，本文聚焦**数据与场景对不上时怎么办**。

## 场景坐标系是前提

DTS 场景使用哪种坐标系由服务端工程配置决定，前端代码无法更改，只能适配：

| 场景类型 | 坐标体系 | 坐标写法 | 典型工程 |
|---------|---------|---------|---------|
| 投影场景（PCS） | 本地平面直角坐标，单位米 | `[x, y, z]` | 工厂、园区、单体建筑 |
| 球面场景（GCS） | 经纬度（WGS84 / CGCS2000） | `[lng, lat, height]` | 城市级、流域级、全球级 |

接入数据前先确认三件事：一，用 `fdapi.coord` 查询当前场景坐标系；二，问清数据方交付坐标系（EPSG 代码，如 CGCS2000 的 4490、Web 墨卡托的 3857）；三，确认高程基准（椭球高还是正常高，两者在国内可差 20~50 米，是"模型悬空/入地"的常见原因）。

## 三类典型偏移与归因

**整体水平偏移几百米到几公里**：大概率是坐标系不一致——例如把 GCJ-02（火星坐标）的点位当 WGS84 用。互联网地图（高德/腾讯）导出的坐标是 GCJ-02，百度是 BD-09，务必先转换为 WGS84/CGCS2000 再入场景（转换方法见[坐标系与坐标转换](/docs/tutorials/coordinates)）。

**整体垂直偏移（悬空/入地）**：高程基准不一致，或数据生产时的基准面与场景不同。可先用固定 z 偏移量校正验证归因。

**局部渐进变形（越往边缘偏得越多）**：投影带选错（如 3 度带与 6 度带混用）或跨带数据未重投影，需回到数据生产端重投影，前端无法根治。

## 加载偏移参数：最后一米的调校

底座数据允许在加载时做平移调校，适合处理"数据本身正确、与场景存在固定偏差"的情况：

```js
// Cesium 3D Tiles 图层支持位置偏移
fdapi.cesium3DTileset.add({
  id: 'city_tiles',
  url: 'https://<tiles-host>/tileset.json',
  offset: [dx, dy, dz],   // 按场景坐标系单位平移
});
```

调校流程建议：在数据覆盖范围的四角与中心各选一个可辨识地物（路口、屋角），用 `fdapi.tools` 拾取场景坐标并与数据坐标对比。**五点偏差一致 → 用 offset 平移解决；偏差不一致 → 是投影/变形问题，回数据端重投影**，不要试图用 offset 硬凑。

## 业务点位的批量配准

从业务系统接入的点位（设备台账、GPS 轨迹）建议在接入层统一做一次坐标清洗，而不是散落在各处转换：

```js
// 接入层统一清洗：GCJ-02 → WGS84 →（投影场景再转平面坐标）
function normalize(points) {
  return points.map(p => {
    const [lng, lat] = gcj02ToWgs84(p.lng, p.lat); // 见 coordinates 教程
    return { ...p, coordinate: [lng, lat, p.height ?? 0] };
  });
}
```

## 配准检查单

- [ ] 已用 `fdapi.coord` 确认场景坐标系（投影/球面）
- [ ] 数据方已提供明确的 EPSG 代码与高程基准
- [ ] 互联网来源坐标已做 GCJ-02/BD-09 纠偏
- [ ] 用 5 个已知地物点验证过偏差的一致性
- [ ] 偏移调校用 offset 参数记录在代码中，而非改动原始数据
